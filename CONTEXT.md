# Vigil — Domain Context

## Glossary

### User
An authenticated person using Vigil. Each User has a unique **email** and **password** credentials used for **sign-in**. **Public self-service registration** is **disabled** for now — there is no open sign-up API or UI. **New Users** are created only through **operator-run tooling** (a CLI command or script shipped with the repo), not through an HTTP registration surface. The same tooling supports **operator password reset** for an existing User (identified by normalized **email**), applying the same **v1 password complexity policy** as initial creation and **change password**; a reset **invalidates every** **Session** for that User so each browser must sign in again. **Email** is **normalized** for storage and comparison: leading and trailing whitespace is removed, and the address is treated as **case-insensitive** (conventionally **stored in lowercase**) so variants that differ only by case or surrounding spaces refer to the same **User**. There is **no display name** in v1 — where the product needs a label for who is signed in, it uses the User’s **email** (normalized form). For **SMS** **Watch notifications**, a **User** may optionally register a **phone number** (E.164) used only for SNS SMS subscriptions; it is not used for sign-in. Passwords set when a **User** is first created and passwords chosen via **change password** must satisfy a **v1 complexity policy**: **at least 12 characters** plus **required character classes** (upper case, lower case, digit, and symbol). **Sign-in** failures (unknown normalized email or wrong password) use a **single generic client-visible message** so callers cannot infer whether an email is already registered. In v1 there is **no email verification** when a User is created — after **sign-in**, the User may use Watches immediately. A User who is signed in may **change password** in v1 (current password plus new password), for rotation or strengthening credentials; a successful change **invalidates every other** **Session** for that User but **keeps the current Session** so that browser remains signed in. This does **not** replace email-based recovery, which remains out of scope for v1. **Forgot-password** and other **email-based self-service account recovery** are **out of scope for v1** (no outbound transactional email requirement for auth); a User who loses their password relies on **operator password reset** via that tooling (or ad-hoc database access), or a future self-service release. Each **Watch** belongs to exactly one User; **Watch Run** data is reached only through that Watch, so another User cannot see another User’s watches or runs. **Sharing** (if ever added) is out of scope for the current model.

### Session
A period in which a **User** is recognized as signed in from a given browser. Established at successful sign-in and cleared at **sign-out**. A User may have **several concurrent Sessions** (e.g. phone and laptop); signing out affects **only** the Session on that browser unless a future “sign out everywhere” feature exists. The client does not store an access token in JavaScript-accessible storage for API auth: it relies on an **HTTP-only cookie** that references **session state held on the server** (opaque id in the cookie, details in the datastore). Vigil v1 assumes the **SPA and API are the same site** (credentials sent only to that site, as in the current dev and Caddy layouts); **CSRF** risk is mitigated with **`SameSite`** on the session cookie and **`Secure`** in production, **not** with a per-request synchronizer or double-submit token — a choice to **revisit** if the SPA and API are ever hosted on **different origins**. In v1 there is **no idle timeout** — absence of activity does not end a Session; it ends when the User signs out on that browser (or when the server invalidates that session through a future explicit mechanism).

### Watch
The central domain concept. A Watch is owned by exactly one **User** and is a user-configured unit of observation: a target URL, an extraction definition, a condition, and a schedule. Vigil periodically fetches the target, extracts a value, evaluates the condition, and records the result. A Watch is either HTML-type (extraction via CSS selector) or JSON-type (extraction via JSONPath). **Watch** APIs are **owner-scoped**: a **Session** may only access watches owned by that **User**; a valid watch id that belongs to another User is answered with **HTTP 404** (indistinguishable from an unknown id for that caller), not **403**. **Cutting over** an environment from a pre-auth build to multi-user v1 **does not** include migrating previously unattributed watches: the operator is expected to start from **no watch rows** (or to clear watch data deliberately); automatic backfill of ownership is out of scope for v1.

### Target
The URL a Watch fetches — either a webpage (HTML) or an API endpoint (JSON).

### Extractor
The rule that pulls a specific value out of a Target's response. For HTML Targets: a CSS selector. For JSON Targets: a JSONPath expression. When configuring a JSON Watch in the client, the form includes a local evaluator: the user pastes sample JSON and sees the extracted value (and a secondary list when multiple nodes match), using the same first-match string extraction behavior as execution. Sample text is not persisted across reloads.

### Condition
The comparison rule evaluated against the extracted value each time a Watch runs. Supported operators: equals, changed, less than, greater than. **Equals**, **less than**, and **greater than** use a fixed **expected value** supplied by the user (string equality for equals; numeric threshold for less than / greater than — interpreted leniently as a floating-point literal, including scientific notation, with surrounding whitespace ignored; the threshold must be a **finite** number when the Watch is saved). **Changed** compares the current extracted value to the previous run's extracted value and does not use an expected value — **expected value must be null** when the operator is changed (non-null values are rejected when the Watch is saved). For less than and greater than, if the extracted value cannot be interpreted as a number at run time, the condition is **not met** (the run still succeeds; there is no error solely for non-numeric extraction). When a Condition is met, the result is recorded in the Watch Run. **Watch notification** may also fire on that outcome according to the rules in that term; the dashboard remains the always-available view of history.

### Watch Run
A single execution instance of a Watch. Records start time, end time, **RunStatus** (`PASS` = completed without execution error, `ERROR` = fetch, extraction, or another execution step failed), the extracted value, and **`conditionMet`**: whether the Condition evaluated to **true** for that cycle (when evaluation ran). **RunStatus must not be read as “condition satisfied” or “condition violated”** — that meaning lives in **`conditionMet`**, interpreted with the Watch’s operator and expected value (especially for less than, greater than, and changed). Stored in Postgres via pg-boss.

### Watch notification
Optional **outbound alerts** configured **per Watch**. A **User** may enable **email** and/or **SMS** independently per Watch (defaults: both **off**). Delivery uses **AWS SNS** in the deployment region: each **User** has SNS **subscriptions** (email to their normalized **sign-in email**; SMS to a **phone number** they provide) that must reach **confirmed** status before alerts are sent on that channel. **SNS subscription confirmation** is required by AWS and is **distinct** from sign-in email verification. **Users** manage endpoints in **account settings** (phone number, confirmation status per channel). Enabling **email** or **SMS** on a **Watch** **also** triggers **`Subscribe`** when that channel’s subscription is missing or not yet confirmed. The server stores each channel’s SNS **subscription ARN** and **status**; alert delivery **`Publish`**es to the **confirmed** subscription **`TargetArn`** for that **User** (not a blind fan-out on the shared topic). **Confirmation status** is refreshed by calling SNS **`GetSubscriptionAttributes`** when the **User** opens notification settings (**poll on demand**) and when they explicitly request a refresh (for example an **“I’ve confirmed”** control); there is no reliance on an inbound SNS HTTP callback to Vigil for v1. **SNS topic** ARN is deployment-level (operator/Terraform). There is **no per-Watch override** destination in v1: email alerts use the **sign-in email** subscription; SMS alerts use the **User**’s configured phone subscription. **SMS:** a **User** must **save a phone number** in account settings before **`notifySms`** can be enabled on any **Watch**; alerts are only **sent** once the SNS SMS subscription is **Confirmed** (UI shows pending until then). **Email:** a **User** may enable **`notifyEmail`** on a **Watch** while the sign-in-email SNS subscription is still **PendingConfirmation** (subscribe is triggered from settings or the Watch flow); **no email alerts are sent** until **Confirmed**; the UI shows a pending state until confirmation completes. **Notification transport** is **deployment-supplied** (**SNS topic**, IAM, region); **Users** do not bring AWS credentials. Alert **content** is **plain text** (subject/body composed by Vigil); SNS may wrap or format messages per channel — the product accepts SNS delivery semantics rather than custom **From** / **Reply-To** control (unlike SMTP). If SNS is **not configured** at deployment level, or a **User**’s needed subscription is **not confirmed**, the **client** must **surface** that alerts **cannot be delivered** on that channel so **Users** are not misled. **User**-facing **delivery history** remains **out of scope** for v1. **Send attempts** on a **notification trigger** (**edge**) are **enqueued** and processed with **pg-boss**, with **bounded retries** for **transient** SNS failures; after exhaustion, the attempt is **abandoned** and **logged** for the **operator**. When the job runs, the worker **re-checks** that the **Watch** still exists and the relevant per-Watch flags are still enabled; otherwise it **skips** without a **User**-visible error. **At-most-one** successful delivery per **edge** per **Watch Run** per enabled channel. **Watch Run** history remains authoritative for observation outcomes. **Edge** rules are unchanged: (1) **Condition met (edge)** — `conditionMet` true after prior run was not true; (2) **Execution error (edge)** — `ERROR` after prior run was not `ERROR`. Message copy includes **Watch** name, which **edge** fired, **time**, extracted value or error excerpt, and a **link** to the **Watch** in the SPA using **`PUBLIC_BASE_URL`**. Fixed framing copy is **English only** in v1. **Local development:** with no **`SNS_TOPIC_ARN`**, notifications are **disabled** (`notificationsReady` false). With **`NOTIFICATION_BACKEND=mock`**, composed alerts are **logged** only (no AWS calls); production and homelab use the real SNS backend.

### Schedule
The cron expression that determines how frequently a Watch runs. Defined per Watch by the user. Minimum granularity: 5 minutes.

## Architecture

Vigil is a pnpm workspaces monorepo with two packages:

- **`apps/client`** — Vite + React 19 SPA, TanStack Router, shadcn/ui
- **`apps/server`** — NestJS + TypeScript, REST (`/api/*`), Prisma, pg-boss

Without a valid **Session**, the client only offers routes required for **sign-in**, plus any **explicitly listed** public routes added later (for example legal notices). All **Watch** UI and the APIs that read or mutate **Watch** / **Watch Run** data require an authenticated **Session**; visiting anything else while unauthenticated should land the visitor on **sign-in** (or an equivalent gate), not a browse-only dashboard. **Sign-in** is **rate-limited in the server application** (not only at the reverse proxy): excessive attempts from the same client identity (for example IP, and optionally the same normalized email) yield **HTTP 429** until the window resets.

The client communicates with the server via a REST API under `/api/*`. TanStack Query wraps a small `fetch`-based client; request/response types are defined twice — once on the server as NestJS DTOs (with `class-validator` decorators) and once on the client as hand-written TypeScript types. Drift is managed manually. On the watch detail route, the client refetches that Watch’s **run history** on a fixed interval (5 seconds) **only while the browser tab is visible** — not the Watch configuration on that interval — so new **Watch Run** rows show up without manual refresh. Run history is loaded in **pages of 20** (**offset/limit**, newest `startedAt` first), with the **current page** carried in **URL search params** so refresh, back/forward, and shared links keep the same slice. Pagination uses **1-based** page numbers (**page 1** = the newest slice). The watch detail route exposes the current slice in the **`runsPage`** URL search param; the runs list API uses a **`page`** query parameter (1-based) and a **fixed page size of 20** (no client-supplied `limit`). Each fetch returns **`{ runs, totalCount }`**: **`runs`** holds the **Watch Run** rows for that page, and **`totalCount`** counts all runs for that Watch so the client can render “page *x* of *y*” and detect an out-of-range page. If the requested page is past the end, the API returns **no rows** for that request; the **client** reconciles the URL to a valid page (last page, or page **1** when there are no runs) using **`totalCount`**, without the server substituting a different page of data. Because new runs are always newer, page 1 stays intuitive; deeper pages can shift slightly while the list grows. After a failed poll, the next attempt is still on that same cadence (no backoff). While run-history requests are in an error state, the watch detail UI shows a short **inline error** in the Run history area and keeps showing the **last successful** run list when one exists; recovery is **automatic** on the same interval (no separate retry control).


Postgres is the only datastore — used for both application data (via Prisma) and Watch scheduling/history (via pg-boss).

### Installation (operator)
The **operator** deploys Vigil on their own **Kubernetes** cluster (intended: a homelab cluster) using a single **umbrella Helm chart** for the application and database. **Postgres** runs **in-cluster** under **[CloudNativePG](https://cloudnative-pg.io/)** (CNPG): the umbrella chart depends on the [CNPG cluster chart](https://github.com/cloudnative-pg/charts) only; the **CNPG operator** is a **cluster prerequisite** already installed on the homelab (not owned by the Vigil release). **Outbound Watch notifications** use **AWS SNS** in **`us-west-2`** (email and SMS via SNS subscriptions — not SES/SMTP). Terraform provisions the **SNS topic**, **`sns:Publish`** / subscription IAM for the app, and related Secrets Manager entries. Sensitive runtime configuration — including **`DATABASE_URL`** and AWS credentials for the notification worker — lives in **AWS Secrets Manager**, provisioned with **Terraform** (intended scope: **Vigil application secrets**, **Route53** records for the public hostname, and **IAM** needed for **cert-manager** DNS-01 and **External Secrets**). **Terraform is the source of truth** for the application database password: it generates the credential, writes **`DATABASE_URL`** to Secrets Manager, and the CNPG cluster is bootstrapped with the **same** password (application database and role **`vigil`**; in-cluster host via the CNPG read-write **Service** DNS name). The homelab runs **k3s**; **[External Secrets Operator](https://external-secrets.io/)** uses a **ClusterSecretStore** backed by an **IAM access key** (installed once as a bootstrap Kubernetes Secret) to sync Secrets Manager values into namespace **Secrets** for the Vigil Helm release. Non-secret config (`PUBLIC_BASE_URL`, `AWS_REGION`, `SNS_TOPIC_ARN`, etc.) is set via Helm values. **Helm values must not contain credentials.** **Infrastructure-as-code** for this installation lives in the **same repository** under `deploy/terraform/` (AWS + k3s bootstrap) and `deploy/helm/vigil/` (umbrella chart). **Terraform** installs **External Secrets Operator** and **cert-manager** on k3s when absent, provisions AWS/Route53/IAM/Secrets Manager, and bootstraps the ESO AWS credential; **Helm** installs the CNPG cluster and Vigil application. **Database schema** is applied by a **Helm hook Job** running **`prisma migrate deploy`** on install/upgrade (not a manual post-deploy step). The production **container image** (Caddy + Nest + built SPA) is built by a **GitLab CI/CD pipeline** and pushed to **Amazon ECR**; the Helm chart pins an **immutable image tag** (not `:latest` in production). **GitLab** authenticates to AWS via **OIDC** → **IAM role** for **ECR push** (no long-lived AWS keys in CI variables). **k3s** uses a **separate read-only IAM principal** for **ECR pull**; a **dedicated credential refresher** (for example a CronJob) keeps the cluster **`imagePullSecret`** valid as ECR tokens expire. **Terraform** provisions ECR, IAM, and related secrets; it does not rely on a one-shot pull credential without refresh. **CNPG** **automated backups** use **Barman** to a **dedicated S3 bucket** in **`us-west-2`** (bucket and backup IAM provisioned in the same Terraform root as Vigil); the umbrella chart enables scheduled base backups and WAL archiving. Restore remains an operator procedure for homelab v1. **Deploys** to k3s are run from **GitLab CI** via the **GitLab Kubernetes agent** (already installed on the homelab cluster): on **version tags only**, the pipeline builds the image, pushes to **ECR**, and runs **`helm upgrade`** with that immutable tag. The Vigil release runs in a dedicated **`vigil`** **namespace**. **Users** reach the product at a single **canonical HTTPS origin**: **`https://vigil.brianwobst.com`**. A cluster **Ingress** terminates **TLS** (intended: **Let’s Encrypt** via **cert-manager**) and forwards all traffic to **Caddy** in the application pod, which routes `/api/*` to NestJS and serves the static SPA for other paths — preserving the same-site **Session** cookie model without CORS. **`PUBLIC_BASE_URL`** must match that origin. **Tailscale** is for **operator** access to the cluster (kubectl, admin); the **Vigil UI is intended to be reachable on the public internet** at the canonical hostname (not via Tailscale Funnel as the primary edge). **DNS** for `brianwobst.com` is hosted in **Route53**; **TLS** is intended via **cert-manager** with **Let’s Encrypt**, using a **DNS-01** challenge against Route53 (homelab-friendly vs port-forward HTTP-01). The CNPG database runs as a **single-instance** cluster (`instances: 1`) with persistent storage — no in-chart HA replica for v1.

Each Watch owns its Schedule. When a Watch runs, pg-boss records a Watch Run with the extracted value, **`conditionMet`**, and **RunStatus** (execution outcome, not condition outcome). The client can trigger a Watch off-schedule via `POST /api/watches/:id/run`, which enqueues a pg-boss job and returns 202 — the same execution path as scheduled runs.

In production, Caddy sits in front of both, routing `/api/*` to NestJS and all other traffic to the static Vite build. In dev, Vite proxies `/api/*` to NestJS so the client uses relative URLs in both environments. No CORS configuration required.

## UI System

Vigil uses **shadcn/ui** (New York style) built on top of Tailwind CSS.

- **Base color**: zinc
- **Accent color**: emerald
- **Border radius**: rounded (default, `0.5rem` base)
- **Style variant**: New York

All UI components must use shadcn primitives where available. Raw Tailwind utilities are permitted for layout and spacing only — not for recreating components that shadcn already provides.

### Theme (appearance)

The client offers **light**, **dark**, and **system** appearance. **System** follows the OS/browser `prefers-color-scheme`. The user’s explicit choice (`light` | `dark` | `system`) is **persisted in `localStorage`**; the **default for a first-time visitor is `system`**. If persistence is unreadable or the value is not one of **`light`**, **`dark`**, or **`system`**, treat the **stored** preference as **`system`** for resolution and for the menu’s **stored**-preference indicator, and **persist `system`** when storage is writable (**self-heal**) so the key matches behavior. **Before the app bundle runs**, a **small inline script in `index.html`** applies the **effective** appearance (read storage, resolve **system** with `matchMedia`) by setting state on **`<html>`** (e.g. `class`) so **first paint** matches preference and avoids a wrong-theme flash. While the stored preference is **system**, **effective** light/dark **updates when `prefers-color-scheme` changes** (no reload required). The document root **advertises `color-scheme`** (`light` or `dark`) to match the **effective** appearance so native controls and scrollbars stay visually consistent with the app chrome. Theme is **changed from the global header on the right** (same chrome on every route): a **compact button** opens a **menu** listing Light, Dark, and System with a clear indicator of the **stored** preference; the **System** row includes a **muted secondary hint** (e.g. that appearance follows the device) so the option is understandable without jargon. The UI is **coherent**: shared surfaces (shell, routes, components) use **semantic color tokens** so the whole app tracks the effective appearance together — not a mix of theme-aware and permanently light-only panels.
